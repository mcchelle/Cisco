Overview

The Problem

Engineering teams and service operations lacked a unified view of delivery velocity, SLA performance, and emerging risk. Cycle time degradation went undetected for weeks, SLA breaches were reactive rather than preventable, and leadership had no single source of truth for operational health.

This project addresses those gaps by building a comprehensive analytics platform that monitors 5,529 pull requests across 7 engineering teams and 10,871 service requests across 10 categories, with a predictive layer that flags at-risk requests before they breach SLA.

Data Layer

Two primary datasets stored in a relational database: GitHub pull request data (5,529 rows across 7 teams and 21 repos) and service request data (10,871 rows across 10 categories and 4 priority levels). Deliberate anomaly patterns were injected, including team degradation trends, holiday volume surges, and staffing bottleneck signatures, to test the detection capabilities of the downstream analytics.

Python SQLite Snowflake-compatible

SQL Transformation Layer

Eight production-grade SQL queries handle all data transformation: weekly cycle time aggregation with LAG-based delta detection, risk signal identification using rolling window functions, SLA compliance rates with conditional aggregation, backlog trend analysis, resolution time distribution with PERCENTILE_CONT, ML feature engineering, and escalation pattern cross-tabulation.

SQL CTEs Window Functions PERCENTILE_CONT

Predictive Modeling Layer

Three models built entirely from scratch using NumPy (no scikit-learn): Logistic Regression with L2 regularization and gradient descent, a Decision Tree using recursive Gini impurity splitting (max depth 8), and a Random Forest (10 bagged trees with bootstrap sampling). An ensemble model averages the Logistic Regression and Random Forest probabilities. Temporal train/test split (80/20) prevents data leakage.

Data Pipeline

SQL Transformation Examples

The SQL layer demonstrates Snowflake-compatible query patterns including CTEs, window functions, and temporal aggregation. Below is a representative query for weekly cycle time analysis with trailing period comparison.

-- Weekly Cycle Time by Team with Trailing Average Delta
WITH weekly_metrics AS (
  SELECT
    team,
    DATE_TRUNC('week', opened_at) AS week_start,
    AVG(cycle_time_hours) AS avg_cycle_time,
    COUNT(*) AS pr_count,
    AVG(CASE WHEN is_merged = 1 THEN 1.0 ELSE 0.0 END) AS merge_rate
  FROM github_pull_requests
  GROUP BY team, DATE_TRUNC('week', opened_at)
)
SELECT
  *,
  LAG(avg_cycle_time, 4) OVER (
    PARTITION BY team ORDER BY week_start
  ) AS trailing_4wk_avg,
  ROUND(
    (avg_cycle_time - LAG(avg_cycle_time, 4) OVER (
      PARTITION BY team ORDER BY week_start
    )) / NULLIF(LAG(avg_cycle_time, 4) OVER (
      PARTITION BY team ORDER BY week_start
    ), 0) * 100, 1
  ) AS delta_pct
FROM weekly_metrics
ORDER BY team, week_start;

Why this matters: This query pattern identifies teams with degrading cycle times weeks before it would appear in a monthly report. The LAG window function compares each week against the same team's performance 4 weeks prior, surfacing gradual degradation that averages would hide.

Business Impact

What This Delivers

Proactive Risk Detection

The predictive model shifts SLA management from reactive to proactive. Instead of discovering breaches after they happen, ops teams receive early warning signals at intake time, enabling resource reallocation and priority adjustments before deadlines are missed.

Automated Health Monitoring

The Team Health classification (HEALTHY / WATCH / AT RISK) eliminates the need for manual weekly reviews. Cycle time degradation is surfaced automatically through trailing period comparison, catching issues like the Infra-SRE degradation trend weeks before it would appear in a monthly report.

Data-Driven Capacity Planning

The backlog trend analysis and PR volume heatmap provide evidence-based inputs for staffing and sprint planning. Quarter-end volume spikes are now quantified and predictable, enabling pre-emptive resource allocation rather than reactive firefighting.

Cross-Functional Transparency

A single dashboard unifies engineering delivery metrics and service operations KPIs, breaking down the silos between teams. The escalation pattern matrix identifies systemic process issues (Vendor Onboarding, Security Review) that require cross-functional solutions rather than team-level fixes.
